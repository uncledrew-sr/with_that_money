# Risk Management (FE)
### [1] 기술적 리스크 (Technical Risks)
1. 물리 엔진(Matter.js) 연산 정밀도 저하
- 리스크 설명 : Matter.js 엔진 내부 연산 시 위치 및 속도 값이 정수형으로 반환되거나 강제 형변환되어 정밀도가 떨어짐. 이로 인해 버블이 화면 밖으로 이탈하거나 겹친 상태로 멈추는 '연산 교착 상태' 발생.
- 리스크 범주 : 기술 / 품질
- 리스크 분석(영향) : 사용자 경험(UX) 저하 및 시각적 오류 발생
- 완화(Mitigation)
    - 렌더링 보정 : 화면 표시 시에는 toFixed(1)을 사용하여 가독성을 높이되, 엔진 내부 연산은 부동 소수점을 유지.
    - 프레임 독립화 : Matter.Runner를 활용해 브라우저 주사율과 독립적인 업데이트 루프 생성.
    - 이탈 방지 : afterUpdate 이벤트를 통해 컨테이너 이탈 감지 시 Body.setPosition으로 위치 강제 재배치.

2. 입력값 미검증에 따른 브라우저 과부하
- 리스크 설명 : 금액 입력에 대한 임계값이 없어, 고액 입력 시 수천 개의 물리 객체(DOM)가 동시 생성됨. 이로 인해 메모리 점유율 급증 및 브라우저 탭 크래시 발생 가능성 있음.
- 리스크 범주 : 기술 / 성능
- 리스크 분석(영향) : 애플리케이션 사용 불가 상태 초래
- 완화(Mitigation)
    - 입력 임계값 설정 : 최소 10,000원 ~ 최대 2,000,000원으로 입력 범위 제한.
    - 객체 생성 제한(Clamping) : getBubbleCount 함수 내에서 렌더링되는 버블 개수가 최대 80개를 넘지 않도록 상한선 설정.
    - 사용자 피드백 : amount-error-msg 요소를 활용하여 제한 범위 이탈 시 즉각적인 에러 메시지 노출.

3. API 서버 연결 및 포트 차단
- 리스크 설명 : 로컬 및 스테이징 환경에서 백엔드 통신 시 방화벽 포트 차단 또는 CORS 정책 위반으로 리소스 로딩 실패. 이로 인해 OAuth 인증 및 핵심 기능(CRUD) 마비 우려.
- 리스크 범주: 기술 / 일정
- 리스크 분석(영향): 프로젝트 핵심 기능 작동 불가 및 개발 일정 지연
- 완화(Mitigation)
    - 환경 변수 중앙화 : config.js를 도입하여 API_BASE_URL을 환경에 따라 유연하게 변경 가능하도록 구조화.
    - 폴백(Fallback) 로직 : ApiService 내 에러 핸들링을 강화하여 통신 실패 시 사용자 알림 제공 및 '경험 모드'로 UI 자동 전환.

4. JWT 인증 만료 및 세션 관리 미흡
- 리스크 설명 : 토큰 만료 시 프론트엔드에서 이를 감지하지 못해 비정상 요청이 지속되거나, 유효하지 않은 정보가 노출될 수 있음. 이는 데이터 보안 리스크로 분류되며 우선순위가 높음.
- 리스크 범주 : 보안 (Security)
- 리스크 분석(영향) : 데이터 무결성 훼손 및 개인정보 노출 위험
- 완화(Mitigation)
    - 인터셉터 구현 : api.js의 handleAuthError를 통해 모든 응답 코드를 감시.
    - 세션 자동 초기화 : 401/403 에러 감지 시 로컬 스토리지 토큰 즉시 삭제 및 window.location.reload()로 강제 로그아웃 처리.
    - 헤더 규격화 : Bearer 스키마 누락 방지를 위한 헤더 생성 함수 표준화.

5. AWS 보안 그룹 차단으로 인한 초기화 데이터 로드 실패
- 리스크 설명 : 로컬 개발 환경(VS Code Live Server, Port 5500)에서 AWS에 배포된 백엔드 서버로 초기 데이터(init 요소)를 요청했으나, AWS 보안 그룹(Security Group) 정책에 의해 5500 포트 접근이 차단됨. 이로 인해 앱 실행 시 카테고리 및 위시리스트 데이터를 불러오지 못함.
- 리스크 범주 : 인프라 / 네트워크
- 리스크 분석(영향) : 프론트엔드 개발 환경에서 API 연동 불가 및 개발 중단
- 완화(Mitigation)
    - 보안 그룹 규칙 수정 : 백엔드 개발자와 협업하여 AWS EC2 인바운드 규칙에 프론트엔드 개발 포트(5500) 허용 설정 추가.
    - CORS 정책 조정 : 백엔드 서버 설정에서 로컬 호스트(http://127.0.0.1:5500) 오리진(Origin)에 대한 리소스 공유 허용.

### [2] 운영 리스크 (Operational Risks)
1. 커뮤니케이션 부재로 인한 개발 지연
- 리스크 설명 : API 스펙 확정 과정에서 명확한 프론트엔드 개발자의 커뮤니케이션 계획 부재로 인해 결과물의 비일관성이 발생함. 이는 프로젝트 일정 지연 및 마감일 미준수 리스크를 초래함.
- 리스크 범주 : 커뮤니케이션 / 일정
- 리스크 분석(영향) : 프론트엔드/백엔드 간 연동 실패 및 재작업으로 인한 생산성 저하
- 완화(Mitigation)
    - 대응 조치 : 마감일 준수를 위해 야간 추가 업무를 수행하여 지연된 일정 만회.

2. 시험 기간으로 인한 노션 협업 일정 미준수
- 리스크 설명 : 시험 기간과 프로젝트 개발 일정이 겹치면서, 사전에 약속된 프론트엔드 개발이 지연됨. 이로 인해 팀 간 작업 현황 공유가 일시적으로 중단되고 팔로우업이 늦어짐.
- 리스크 범주 : 운영 / 일정
- 리스크 분석(영향) : 프로젝트 현황 파악 불가 및 협업 리듬 저하
- 완화(Mitigation)
    - 유연한 일정 관리 : 시험 기간 동안은 필수적인 개발 외의 문서화 작업을 일시 유예(Freeze)하고, 시험 종료 직후 집중 보완 기간을 설정하여 밀린 문서를 일괄 업데이트함.

### 회고
- 문제 해결을 대하는 태도의 변화
    - 이전에는 에러가 발생하면 무작정 구글링을 하거나 AI의 답변을 복사하기 바빴습니다. 하지만 이번 프로젝트에서는 백엔드 팀원과 함께 "현상 파악 → 가설 나열 → 검증"의 단계를 밟으며, 왜 이 수치가 정수여야 하는지, 왜 이 포트가 막혔는지에 대한 근본적인 원인을 고민하게 되었습니다.

- 시스템적 사고의 확장
    - 프론트엔드는 단순한 화면 구성이 아니라, 백엔드에서 설계한 아키텍처 위에서 데이터가 의도한 대로 흐르게 하는 파이프라인의 종착점임을 깨달았습니다. 특히 JWT 만료 대응을 위해 인터셉터 로직을 구현하면서 보안과 인증의 흐름을 깊이 있게 이해할 수 있었습니다.

- 앞으로의 다짐
    - 물리 엔진의 정밀도 이슈나 인프라 차단 문제처럼 예상치 못한 '삽질'의 시간들이 결국 제 코드를 더 탄탄하게 만들었습니다. 앞으로도 마주할 수많은 난관 앞에서 스트레스받기보다, 내가 만든 시스템의 뼈대를 더 튼튼하게 보강할 기회로 여기며 즐겁게 몰입하는 개발자가 되려고 합니다.
